    # 前言

    Java之前几乎没有语言将目标专注于客服开发与维护程序的复杂性
最终导致项目复杂，难以维护
C++选择向后兼容C，以及具有C一样的高效，但暴露更多的复杂性问题

Visual Basic VB选择与Basic绑在一起，结果建立在VB纸上的所有扩展导致无法维护的语法

perl向后兼容awk sed grep 以及所有它打算代替Unix工具，结果Perl程序变成了不可阅读的代码

设计师们也都为了解决复杂性问题做了某种程度的工作。并且正是解决某类特定问题的能力，成就了它们的成功

Java为程序员减少了复杂性，关心的是减少开发健壮代码所需的时间和困难
早期这个目标使得代码运行并不快，现在Java程序运行效率已经改善了
但它显著缩短了开发时间与C++开发相比，只需一半甚至更少的开发时间
且开始着手各种复杂任务，如多线程，网络编程，并作为语言特性或者工具库的形式纳入Java
且解决了一些相当大的复杂性问题：跨平台，动态代码修改，甚至是安全的议题

    # 对象导论

    # 抽象过程

    OOP Object-oriented Programming 面向对象编程
所有编程语言都提供了抽象机制，可以认为，人们所能解决的问题的发咋行直接取决于抽象的类型(所抽象的是什么)和质量

汇编语言是对底层机器的轻微抽象
所谓的命令式语言如FORTRAN BASIC C等都是对汇编语言的抽象
这些语言在汇编语言的基础上有了大幅的改进，但是所作的主要抽象仍要求基础计算机的结构，而不是基于问题的结构来考虑
程序员必须建立起在机器模型和实际待解问题的模型(如业务)之间的关联。这种映射是费力的，而且这不属于编程语言所固有的功能，使得程序难以编写，且维护代价高昂

另一种对机器建模的方式就是只针对待解决问题建模。
早期的LISP和APL都选择考虑世界的某种特定视图(分别是"所有问题最终都是列表"或"所有问题都是算法形式") 
PROLOG则将所有问题都转换成决策链。此外还产生饿了基于约束条件变成的语言和专门通过对图形符号操作来实现变成的语言(后者被证明限制性过强)
这些方式对它们所要解决的特定的类型的问题都是不错的方案。但一旦超过其特定领域，就力不从心了

OOP雨荨根据问题来描述问题，而不是根据运行解决方案的计算机来描述问题。
每个对象看起来都像一台计算机---它具有状态，还具有操作，用户可以要求对象执行这些操作。

Smalltalk(Java所基于的语言之一)的五个基本特性
1.万物皆为对象
2.程序是对象的集合，通过发送消息来告知彼此所要做的。（对象的方法）
3.每个对象都有自己的由其他对象所构成的存储
可以通过创建包含现有对象的包的方式来创建新类型对象。因此，可以在程序中构建复杂的体系，同时将复杂性隐藏在对象的简单性背后
4. 每个对象都有其类型Class. 每个对象都是某个class的实例instance
每个类区别于其他类的特性就是：可以发送什么样的消息给它
5. 某一特定类型的所有对象都可以接收同样的消息

因为类描述了具有相同特性(数据元素)和行为(功能)的对象集合
实际上就是一个数据类型
浮点型数字具有相同的特性和行为集合
差异在于，程序员通过定义类来适应问题，而不是被迫只能使用现有的用来表示机器中的存储单元的数据类型

类型Light 接口 on() off() brighten() dim（）
那么Light的实例对象就必须满足这些接口

    # 服务提供者

    在试图开发或理解一个程序设计时，最好的方法之一就是将对象想象为服务提供者。
"如果可以将问题从表象中提取出来，那么什么样的对象可以马上解决我的问题呢？"
ex. 簿记系统
1.预定义的簿记输入屏幕的对象
2.执行簿记计算的对象集合
3.一个处理在不同打印机上打印支票和开发票的对象
某些可能已经存在了，那么对于不存在的对象，它们看起来像什么？能提供哪些服务？需要哪些对象才能旅行它们的义务？
有助于提高对象的内聚性。
以上是高内聚
如果将过多的功能都塞在一个对象中。就出现高耦合
高内聚低耦合是高质量代码的基本要求之一。
ex. 在检查打印模式的模块中，你想让它了解所有的格式和打印技术。
你可能需要三个甚至更多个对象。
1. 所有可能的支票排版的目录，它可以被用来查询有关如何打印一张支票的信息
2. 一个通用的打印接口，知道所有不同类型的打印机的信息
3. 第三个对象通过调用另外两个对象的服务来完成打印任务
这样每个对象都有一个它所能提供服务的内聚的集合。
每个对象都可以很好的完成一项任务，但是它并不试图做更多的事。

    # 被隐藏的具体实现

    将开发人员按角色分为类创建者和客户端程序员（消费者）
类的创建者的目标就是构建类，这种类只像客户端程序员暴露必须的部分，而隐藏具体实现
理由是，如果隐藏，那么消费者就不能访问。同时意味着创建者可以任意修改隐藏的部分，而不必担心对使用者造成影响
同时一般隐藏的部分是脆弱的部分可以减少bug

访问控制：暴露必须的，隐藏具体实现
关键字 public private protected

    # 复用

    产生一个可复用的对象设计需要丰富的经验和敏锐的洞察力
一旦类创建并测试完，在理想的情况下就代表一个有用的代码单元。
但事实上，这种复用性并不容易达到所希望的那种程度

最简单的复用就是直接使用类的实例对象
也可以将某个类的一个对象放置于某个新的类中。称为“创建成员对象”，是一种称为组合composition的概念

如果组合是动态放生的，则通常被称为聚合aggregation

组合经常被视为has-a关系。如汽车拥有引擎
组合带来了极大的灵活性。新类的成员对象通常都是private

继承不具备这样的灵活性，因为编译器必须对通过继承而创建的类施加编译时的限制

在建立新类时，优先考虑组合，因为更加简单灵活。

    # 继承

    创建一个类之后，即使另一个新类与其具有相似的功能，你还是得重新创建一个新类

可以通过继承，以现有的类为基础，复制一份，通过添加和修改来修改这个副本
注意：实际上并不是副本，而是引用。基类/源类/超类/父类发生变动时，会影响继承类/导出类/子类也会反映这些变动

Shape : draw（） erase()
Circle Square Triangle extends Shape
继承以后，即便子类里没有定义方法，它可以引用到父类的方法

两种方法可以使得基类和导出类产生差异
1.在导出类中添加新方法
Triangle flipVertical()
这意味着基类不能直接满足你的所有需求，因此必须添加更多的方法。
这种简单而基本的使用方式，有时对问题来说确实是一个完美的解决方式

第二种改变基类的方法的行为，称之为覆盖overriding
在导出类中定义与基类相同得方法
可以说：我正在使用相同得接口，但是想要在新类中做些不同的事情
Circle draw()
Square draw()
Trangle draw()
子类中有就优先调用子类的


is-a 和 is-like-a
某种争论：继承应该只覆盖基类的方法吗
1. 如果只覆盖基类的方法，就意味着导出类和基类是完全相同得类型，因为它们具有完全相同得接口。
结果可以用一个导出类对象来完全代替一个基类对象。
可以视为纯粹代替，通常称为替代原则
某种程度上，是一种处理继承的理想方式
is-a关系：圆形是一个几何形 

有时候必须在导出类型中添加新的接口元素。这个新类型仍然可以代替基类，但是这种代替并不完美。因为基类无法访问新添加的方法。
is-like-a关系
新类具有旧类型的接口，但是它还包含其他接口

Cooling System制冷系统 cool()
Air Conditioner ,Heat Pump extends Cooling System
Air Conditioner cool()
Heat Pump cool() heat()
看完这个后，会显然发现CS这个基类不够一般化。应该改为温度控制系统，是它包括制热功能。这样我们就可以套用替代原则了

    # 多态

    站在子类角度是继承
站在父类角度是多态

在处理类型时，经常想把一个对象不当做它所属的特定类型来对待，而是将其当做基类对象来对待。
使得可以写出不依赖于特定类型的代码。
几何形中定义的方法，不需要考虑实际是正方形还是三角形

BirdController reLocate()
Bird move()
Goose move()
Penguin move()

Penguin, Goose extends Bird
BirdController对象仅仅处理泛化的Bird对象，而不需要知道具体是Goose还是Penguin
当调用move()时发生了什么。move调用时，即便忽视Bird的具体类型，也会正确调用Goose的飞 Penguin的走呢

答案：编译器不可能产生传统意义上的函数调用。

非面向对象编程的编译器产生的函数调用会引起所谓的前期绑定。
这意味着编译器将产生对一个具体的函数名字的调用，运行时，将这个调用解析到要被执行的代码的绝对地址。

然而在OOP中，程序运行时，才能确定代码的位置。
采用后期绑定的概念。
编译器确保被调用方法的存在，并对调用参数和返回值进行类型检查。
但是并不知道被执行的确切代码
无法做类型检查的语言成为弱类型语言

为了执行后期绑定，Java使用一小段特殊的代码来代替绝对地址调用。（第八章）
这段代码使用在对象中存储的信息来计算方法体的地址。
这样，根据这一小段代码的内容，每一个对象都可以具有不同的行为表现。当向一个对象发送信息时，该对象就能够知道对这条消息应该做什么

在某些语言钟。如C++是通过virtual关键字实现后期绑定
Java默认后期绑定

正因为多态
void doSomething(Shape shapre){
	shapre.erase() //真正调用的时候circle 和 line又是不一样的
}

doSomething(circle)
doSomething(triangle)
doSomething(line)
Circle Triangle Line 类型是Shape的子类，可以代替Shape作为参数放入
把子类当做父类，叫向上转型upcasting

    # 单根继承

    基本上所有OOP语言除C++，所有类最终都继承单一基类。Object
所有对象都具有一个共用接口

C++多继承，能够更好地适应C模型，而且受限较少。
但要进行完全的OOP时，必须构建自己的继承体系，使得它可以提供其他OOP语言内置的便利。并且在任何新类库中，总会用到一些不兼容的接口，需要花力气有可能通过多继承来使得新接口融入你的设计之中。
这么做来换取C++额外的灵活性是否值得呢？
如果在C上面投资巨大的话，这么做就很有价值。
如果从头开始Java这样的选择通常有更高的生产率

单继承使得垃圾回收器的实现变得容易得多。
垃圾回收器是Java相对C++的重要改进之一
且由于队友对象都保证具有其类型信息，因此不会因为无法确认对象的类型陷入僵局。这对于系统级操作如异常处理显得尤其重要。

    # 容器

    通常来说，如果不知道解决特定问题需要多少个对象，或者它们将存活多久，那么就不可能知道如何存储这些对象。
如何才能知道需要多少空间来创建这些对象呢？---不可能知道，因为只有运行时才能获得。

这个问题的解决方案似乎显得轻率：创建另一种对象类型。
这种类型持有对其他对象的引用。
当然也可以用在大多数语言钟都有的数组类型来实现相同的功能。

通常这个类型被称为容器。也称为集合。
在任何需要时都可以扩充自己以容纳你放置的所有东西。

幸运的是，好的OOP语言都有一组容器，作为开发包的一部分。
在C++中，容器是标准库的一部分。
Java在其标准库中也含有大量容器
在某些类库中，一两个通用容器就足够满足所有需求了。
但在其他类库，如Java中，具有满足不同需要的各种类型的容器
List Map set,以及诸如队列，树，堆栈等更多的构件

从设计的观点来看，真正需要的只是一个可以被操作，从而解决问题的序列。
好像没有理由设计不同种类的序列。
但这是需要的
原因1：不同容器提供了不同类型的接口和外部行为
堆栈相比队列就具有不同的接口和行为，也不同于集合和列表
它们之中的某种容器提供的方案可能更加灵活

原因2：不同的容器对某些具体操作具有不同的效率
ex. 两种List ArrayList 和 LinkedList
ArrayList 随机访问元素时是固定时间
LinkedList：随机访问元素时越靠近表尾的元素，花费时间越长，在序列中插入某一个元素时，开销更小

    # 参数化类型 泛型

    SE5之前，容器存储的对象都是通用类型Object。
单继承的结构使得可以存储Object的容器可以存储任何东西

但是因为只能存Object，所以存入时必须被向上转型为Object，因此它会丢失其身份（如Circle），当它取回时，就获取了一个Object对象的引用，而不是Circle类型的对象的引用。
那么如何变换回来呢？

向上转型是安全的
除非确切知道类型，否则向下转型几乎是不安全的
当然也并非彻底是危险的，如果转错了，会得到异常
即便如此，必须要知道原本的类型，才能正确的向下转型
如何做呢？

那么创建一个知道自己所保存的对象的类型，从而不需要向下转型以及消除犯错的可能的容器---这个解决方案交错参数化类型机制

参数化类型机制是一个编译器可以自动定制作用域特定类型上的类。
ex. 可以定制一个只接纳和取出Shape对象的容器

SE5重大变化之一就是参数化类型，Java中成为泛型<>
ArrayList<Shape> shapres = new ArrayList<Shape>()

    # 对象的创建和生命周期

    使用对象时，最关键的问题之一便是它们的生成和销毁方式。
每个对象为了生存都需要资源，尤其是内存。当不再需要时，必须被清理掉。
在相对简单的变成情况下，手动清理看起来似乎不是什么挑战。

但是复杂的情况下，如为机场设计空中交通管理系统
1. 创建一个鞥器保存所有飞机
2. 进入管控区的飞机创建一个飞机对象，放置于容器中
3. 离开此区域后，删除相关的飞机对象

但是可能别的系统也记录着有关飞机的数据，这些数据不需要像上面那样立刻就能注意。
如记录所有飞离机场的小型飞机的飞行计划，你需要第二个容器来存放小型飞机。
无论何时只要创建的是小型飞机的对象，它同时也要被存入这个第二个容器中。
然后某个后台进程在空闲时对第二个容器内对象进行操作。

当处理完某个对象之后，其他部分可能还在处理它。
怎样才能知道何时销毁呢？
在C++，这个问题会变得非常复杂。
C++ 认为效率控制是最重要的。所以给程序员选择的权利。对象的存储空就生命周期可以在编程时确定。
可以将对象放在堆栈或静态存储区内来实现。
这种方式将存储空间分配和释放位于优先考虑的位置，某些情况下这样控制非常有价值。但是牺牲了灵活性。因为必须在编写的时候知道对象确切的数量，生命周期和类型。

第二种方式就是再堆的内存池中动态地创建对象。只有在运行时才能确定需要多少对象。如果需要一个新对象，可以在需要的时候直接在堆里创建。
因为存储空间是被动态管理的，所以需要大量的时间在堆中分配存储空间。
这可能远远大于在堆栈中创建存储空间的时间。

堆栈中创建存储空间和释放存储空间通常各需要一条汇编指令即可，分别将栈顶指针向下移动和栈顶指针向上移动

创建堆存储空间的时间依赖于存储机制的设计

动态方式有这样一个一般性逻辑假设：对象趋于变得复杂，所以查找和释放存储空间的开销不会对对象的创建造成重大冲击。

Java完全采用了动态内存分配方式。通过new关键字创建对象

关于对象生命周期
对于允许在堆栈上创建对象的语言，编译器可以确定对象存货的时间，并且可以自动销毁。

如果是堆上创建对象，编译器就会对它的生命周期一无所知。在像C++这样的语言钟，必须通过编程的方式来确定何时销毁。
常见的内存泄漏问题

Java提供了垃圾回收器机制，可以自动发现对象何时不再被使用，从而销毁它
减少了必须考虑的议题和必须编写的代码。更主要的是表面暗藏的内存泄漏问题。
很多C++死在这里。

    # 异常处理：错误处理

    自从变成语言问世以来。错处处理始终是最困难的问题之一。
因为设计一个好的错误处理机制很困难。很多语言直接略去了这个问题，让程序员设计者处理。
设计者也只是提出一些不彻底地方法，这些方法适用于一些容易避免此错误的场合，而且解决方式通常也只是忽略此问题。
依赖于程序员自身的警惕性。而不是语言所强制的。

异常处理，将错误处理直接置于语言钟，甚至有时在OS中。
异常是一个对象，它从出错地点被抛出，并被异常处理器捕获。
异常处理和程序正常执行路径并行，不干扰正常的执行代码。
使得代码编写变得更加简单。

此外异常不能被忽略，它保证一定会被某处得到处理
提供了一种从错误状态进行可靠恢复的途径。不再是只能退出程序，你可以经常进行校正，并恢复程序的执行。有助于编写出更健壮的程序

值得注意的是，异常处理不是面向对象的特征。在OOP之前就已经存在了，尽管在OOP中尝尝被表示成一个对象

    # 并发编程

    在同一时刻处理多个任务的思想。
许多程序设计都要求，程序可以停下正在做的工作，转而处理某个其他问题，再返回。
很多方法可以实现
最初是根据程序员掌握的有关机器底层的知识来编写中断服务程序，主进程的挂起是通过硬件来触发
但是难度太大，不能移植。移到新型号机器上时，费时费力

有时中断对于处理时间性强的任务是必须的，但是对于大部分的其他问题，我们只是想要将问题切分成多个可以独立运行的部分(任务)，从而提高程序的响应能力。
这些独立运行的任务叫做线程

上线的概念被称作并发。

通常线程只是一种为单一处理器分配执行时间的手段。
如果是多核的话每个任务都可以被指派给不同的处理器，这样才是真正的并行执行。

程序员不用关心机器是不是多核。只要逻辑上是并行处理，如果机器有多核，自动适应

隐患：共享资源。如果并行任务同一时间都要访问同一项资源，那么就会出现问题。
通过锁
某个任务锁定某项资源，完成其任务，然后释放资源锁之后，其他任务才可以使用

    # 用引用操纵对象

    java是基于c++的，但它是一种更纯粹的面向对象程序设计语言
c++和java都是混合/杂合型语言。但java认为这种杂合性并不像c++中name重要。
杂合型语言允许多种编程风格。c++之所以成为一种杂合型语言主要是为了支持C。是C的一个超集，势必包括许多C没有的特性，使得C++在某些方面过于复杂

每种语言都有自己操作内存中元素的方式。
必须注意数据是什么类型。
是直接操纵元素，还是用某种基于特殊语法的间接表示（如c和c++中的指针）

这一些得到简化，因为一切都是对象，因此可采用单一固定的语法。
注意 标识符实际上是对象的一个引用reference.
遥控器(引用)和电视机(对象)
String s = "123"

s是引用 "123"是对象

简单理解成指针也行，不过java中的引用语法书更接近c++的引用而不是指针

通过new关键字来创建一个对象
String s = new String("abc")

存储位置
程序运行时候，对象是怎么进行安排放置的，特别是内存是如何分配的？

1. 寄存器
最快！ 因为它位于处理器内部(与其他存储区不一样)
但是数量有限，所以一般都是按需分配，程序员不能直接控制，在程序中也感觉不到它的存在。
在C和C++中允许你向编译器建议寄存器的分配方式

2. 堆栈
位于RAM
堆栈指针可以从处理器那里直接获得支持。
指针向下移动，分配新内存
向上移动，释放内存
快速有效，仅次于寄存器
但是必须知道存储在堆栈内所有项的生命周期才能便于上下移动堆栈指针。
限制了程序的灵活性
所以对象不能存储在这里。
而对象的引用存在这边

3. 堆
存放对象
不同于堆栈：编译器不需要知道存储的数据再堆里活了多久。因此在堆里分配空间有很大灵活器。需要对象时，new以下。
代价是，用堆进行存储分配和清理，可能比用堆栈需要更多时间

4. 常量存储
通常直接存放在程序代码内部，这么做是安全的，永远不会被改变。
有时嵌入式系统，常量本身会与其他部分分隔开，这时候可以选择存在ROM 只读存储器。

5. 非RAM存储
数据完全存活于程序之外。在程序没有运行时，也能存在。
基本的例子就是流对象和持久化对象中
在流对象中，对象转化成字节流，通常被发送给另外一台机器。
在持久化对象中，对象被存放于磁盘上。
在需要时，可以恢复成常规的，基于RAM的对象。
JAVA提供了对轻量级持久化的支持，而JDBC和Hibernate等提供了更加复杂的对数据库里存储和读取对象信息的支持

特例：基本类型

因为对象存在堆里，因此当对象特别是小的，简单的变量，那么就不是很有效。
对于基本类型，采取跟C和C++的相同方式，不用new来创建，而是将这个变量直接存值，放在堆栈中

java中每种基本类型所占空间大小是一样的。
这个不变性是更可移植性的原因之一。

![这是图片](/assets/imgs/java/%E5%9F%BA%E6%9C%AC%E7%B1%BB%E5%9E%8B.png "基本类型")

包装类使得基本类型，可以在堆中创建一个非基本对象，用来表现对应的基本类型

ex.
char c = 'x'
Character ch = new Character(c)
或
Character ch = new Character('x')

SE5 自动包装，自动将基本类型转为包装器类型
Character ch = 'x'
也可以反转
char c = ch

java中的数组
几乎所有语言都支持数组。在C和C++中使用数组时危险的。
因为它们的数组就是内存块。如果一个程序要访问自身内存块之外的数组，或者在初始化前使用（常见错误），都会产生难以预料的后果。

java目标之一安全性。
确保数组会被初始化。且不能再它范围之外被访问。
这个范围检查，是以每个数组上少量的内存开销以及运行时的下标检查为代价。
换来安全性和效率的提高，因此是值得的。
且java有时可以优化这些操作

创建一个数组，同样实际上就是创建了一个引用数组，每个引用都会自动被初始化一个特定值，该值拥有自己的关键字null。一旦看到nl，java就知道这个引用还没有指向某个对象。在使用任何引用前，必须为其指定一个对象。
如果试图使用一个还是null的引用，那么运行时会报错。因此，常犯的错数组错误在java中就可以避免了。

还可以创建基本数据类型的数组。同样编译器会保证它的初始化。因为它会将数组所占的内存全部置零
